データの流れ — 詳細

入力データフロー（インバウンド）

1
媒体からの応募データ取込
マイナビ・リクナビ等のAPIまたはCSVから応募者情報を定期取込。媒体固有の項目はJSONBフィールドに格納。
API連携 CSVインポート
2
QRコードスキャン → 直接登録
学校訪問・説明会等でQRをスキャン。LIFFアプリが開き、学生が直接プロフィール入力。QRに紐づいたタグが自動付与される。
LINE LIFF + QR
3
マイページ入力（書類・追加情報）
学生がLIFFマイページから書類アップロード・個人情報補足入力。カスタムフィールドにも対応。
LIFF App

出力データフロー（アウトバウンド）

A
LINE一斉送信
タグ・ステータスで絞り込んだ応募者にLINE送信。Messaging API multicast/narrowcastを使用。
LINE Messaging API
B
書類PDF生成
内定通知書・承諾書等をテンプレートから生成。応募者データを差し込み。
C
Salesforce登録（承諾時）
承諾ステータス変更をトリガーにSalesforce REST APIでContactを自動生成。
SF REST API
D
CSVエクスポート
全応募者データ・選考データ・アンケートデータを随時エクスポート可能。

LINE友達追加 — 2パターンの設計

🟢 パターンA：QRコード直接登録
（学校訪問・説明会で推奨）

1
管理者がQR発行
流入経路タグ付きのLIFF URLを含むQRを生成
liff.me/xxxxx?source=tokyo_univ
2
学生がQRをスキャン
LINE内ブラウザでLIFFアプリが起動。LINE自動ログインでユーザーIDが取得される
liff.getContext().userId
3
LIFFで簡易登録フォーム
氏名・学校・メールアドレスを入力（最小限）
4
ATS：応募者レコード生成
LINE UserID + 入力情報 + 流入タグで
新規レコードを作成。友達追加も同時完了
LINE UserIDとATSレコードが同時に生成されるため、紐付け処理が不要。最もシンプル。

🔵 パターンB：媒体経由 → LINE後付け連携
（マイナビ・リクナビ等で応募後）

1
媒体から応募データ取込
ATSに応募者レコードを生成。この時点ではLINE未連携
API / CSV
2
一意トークン発行
応募者ごとにユニークなトークンを生成
UUID or short hash、有効期限あり
3
LINE招待URLを送付
媒体のメッセージ or メールで下記URLを送付
liff.me/xxxxx?token={UUID}
⚠️ メールアドレス必須
4
学生がURLをタップ → LIFF起動
LINE内でLIFFが開き、liff.getContext()でLINE UserIDを取得。tokenとUserIDをバックエンドに送信
LIFF + token照合
5
ATS：LINE UserID紐付け
tokenで応募者レコードを検索 → LINE UserIDを保存。以降はLINEで双方向コミュニケーション可能
既存の応募者レコードに後からLINEを紐付けるパターン。媒体API連携で多用。

custom_fields（JSONB）設計の考え方

固定フィールド（全媒体共通）
{
  "name": "田中 一郎",
  "school": "早稲田大学",
  "email": "tanaka@example.com",
  "graduation_year": 2026
}
custom_fields（媒体・用途ごとに柔軟）
{
  "mynavi_score": 82,
  "club_activity": "サッカー部",
  "toeic": 850,
  "preferred_location": "東京",
  "intern_history": ["〇〇社", "△△社"]
}
媒体ごとに異なる項目・独自の採用評価項目は全て custom_fields (JSONB) に格納。管理画面から項目の追加・表示設定が可能な設計にする。PostgreSQLを前提とすることで検索・インデックスも対応可能。

媒体連携フロー

API連携（自動取込）

1
定期ポーリング or Webhook
媒体側の新着応募を定期取得（例：1時間ごと）
2
フィールドマッピング変換
媒体固有の項目名をATSの共通フォーマットに変換。差分はcustom_fieldsへ
3
重複チェック・UPSERT
メールアドレスまたは媒体IDで同一人物を判定。存在すれば更新、なければ新規作成
4
LINE招待URL自動生成
新規登録時にトークン生成。媒体のサンクスページ or メールで案内

CSVインポート（手動）

1
CSVダウンロード
媒体の管理画面からCSVをダウンロード
2
ATS画面でアップロード
媒体を選択してCSVアップロード。プレビューで確認後、インポート実行
3
マッピング・重複チェック
API連携と同様の処理でレコード生成
API未対応の媒体はCSVで対応。最終的にはAPI連携に移行推奨。

重複排除ロジック

①
LINE UserIDで照合
LINE連携済みなら確実に同一人物と判定
②
メールアドレスで照合
同一メールアドレスが存在すれば更新
③
氏名 + 学校で照合
メール未登録の場合の補助的照合。要手動確認フラグ付与
④
新規作成
いずれにも該当しない場合のみ新規レコード生成

Salesforce → ATS（定期同期）

1
在籍情報の同期
退職・部署異動のレコード変更をSFから定期取得（日次バッチ）。ATSの応募者レコードに在籍履歴として保存
SF SOQL / Bulk API
2
大学情報の取得
SFのAccountオブジェクト（大学）から訪問履歴・担当者情報を取得。ATSの学校マスターと紐付け
SF AccountオブジェクトAPI
3
OB/OG情報としての活用
退職者が訪問・リファラル等で再接点を持った場合、ATS上で在籍履歴付きのOB/OGとして管理
主従関係の設計方針：
採用情報（選考・評価）→ ATSがマスター
在籍・退職・異動情報 → SFがマスター
学校・大学情報 → SFがマスター（ATSに読み取りキャッシュ）

ATS_Applicant_ID__c による双方向リンク

SalesforceのContactに ATS_Applicant_ID__c（カスタム項目）を持たせることで、 どちらのシステムからも対応するレコードを特定可能にする。これがコンフリクト解消の基盤となる。

id: "ats-uuid-xxxx"
sf_contact_id: "SF-003xxxx"
