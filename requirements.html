業務要件（ビジネスゴール）

ID業務ゴール
BG-01新卒採用のエントリーから内定承諾まで、採用プロセス全体を1システムで完結させる
BG-02複数媒体・流入経路からの応募者を一元管理し、情報の分散・二重入力を排除する
BG-03LINEを主要コミュニケーションチャネルとして、学生との接点を最大化する
BG-04エントリー〜承諾の各ステップ遷移率を可視化し、採用活動の改善に活用する
BG-05内定承諾後はSalesforceと連携し、入社後の人材管理へシームレスに引き継ぐ
BG-06自社専用の閉じたシステムとして運用し、外部へのデータ流出リスクを最小化する

システムスコープ

対象ユーザー

採用担当者管理画面全般を操作。メッセージ送信・選考管理・分析
面接官面接評価の入力・閲覧（限定権限）
エージェントカレンダー経由での面接予約調整
応募者（学生）LINEおよびLIFFマイページから操作

応募者区分

新卒大学・大学院在学中。卒業見込み年度あり
既卒卒業後未就業。Re就活等から流入
第二新卒卒業後3年以内の転職者。Re就活から流入の可能性あり
⚠️ 第二新卒含むかは確認中（U-03）

非機能要件

カテゴリ要件
セキュリティ HTTPS必須・認証必須（2要素認証推奨）・アクセスログ記録
自社専用クローズドシステム。外部からのアクセス制限（IP制限またはVPN推奨）
管理者・面接官・閲覧者の権限分離（RBAC）
個人情報保護 個人情報保護法・採用情報の取り扱い規程に準拠
不採用者・辞退者データの保持期間ポリシーを定義（要ヒアリング）
データ削除・匿名化機能を実装
可用性 採用繁忙期（10月〜3月）の安定稼働。目標稼働率 99.5% 以上
計画メンテナンスは深夜帯に実施
性能 学生一覧（500件）の表示：2秒以内
LINE一斉送信（500名）：エラーなく完了（Messaging API のレート制限考慮）
ファネル分析の集計：5秒以内
拡張性 採用人数・担当者数増加時にスケールアウト可能な構成
媒体追加・フロー追加が管理画面から行える設計
custom_fields（JSONB）で任意の応募者属性を追加可能
操作性 採用担当者が直感的に操作できるUI
スマートフォンからの管理操作も考慮（レスポンシブ対応）
LINE LIFF（学生側）はモバイルファーストで設計
データ保全 日次バックアップ。過去3年分のデータ保持を想定
Salesforce との同期ログも保持

ID状態確認事項詳細・決定内容
U-01 確認中 LINE既読確認の代替案合意 個人単位の既読取得はLINE API仕様上不可。マイページアクセス日時を「最終確認日」として代用する方向で調整中
U-02 ✓ 確定 連携媒体名の確定 必須6媒体：マイナビ・Re就活キャンパス・OfferBox・キミスカ・キャリアチケット・ABABAに確定（v0.6更新）。
オプション：ワンキャリア・ホワイト企業ナビ（Phase 2以降で判断）。
※ リクナビはクライアントリストから外れたため除外。
※ 初期はCSV連携から開始。APIパートナー契約の要否は媒体ごとに個別確認が必要。
U-03 確認中 応募者スコープ（新卒のみ vs 第二新卒含む） Re就活が主に既卒・第二新卒向けのため確認が必要。スコープによってデータモデルの卒業年度フィールドや職歴フィールドの扱いが変わる。
→ 柔軟な設計（custom_fields）で対応できるよう準備済み
U-04 ✓ 確定 SF連携の主従関係 Salesforceが主（マスター）。退職・異動データはSF→ATSの一方向。コンフリクト時はSF優先。ATSは読み取り専用キャッシュとして扱う
U-05 UPD 確認中 エージェント利用有無 自社新卒採用のため、外部エージェントを使うケースが少ない可能性あり。
新卒採用はマイナビ・リクナビ等のナビサイトが主流のため、エージェント連携（FR023）は不要かもしれない。
→ 先方に「新卒採用でエージェントを使っているか」を確認する。不要であれば FR023 はスコープ外に。
U-06 未確認 面接評価項目の詳細 評価シートの項目・評価スケール（5段階等）。職種・フローごとに異なる評価シートを使うか
U-07 未確認 説明会の形式 オンライン・対面両方か。Zoom等のURL自動付与が必要か。定員管理が必要か
U-08 UPD 一部確定 担当者数・権限パターン 権限ロールは4種類に確定： 管理者 / 採用担当 / 面接官 / 閲覧のみ
面接官は自分が担当した学生のみ表示（閲覧・評価入力）。
残り確認：採用担当者は何名か。面接官の人数。

懸念事項・リスク

ID深刻度懸念・リスク詳細と対応方針
R-01 高 LINE Messaging API のコスト メッセージ送信数によって月額費用が変動。採用繁忙期に大量送信した場合のコストが想定外になる可能性。
対応： 月間送信数を見積もりプラン選定。フリープラン（200通/月）を超える場合は有料プランへ
R-02 高 LINE 個人既読確認の制約 「学生の既読確認」要件はLINE API仕様で実現不可。要件調整が必要。
対応： マイページアクセス日時を代替指標として提案中（U-01）
R-03 高 媒体API連携の難易度・コスト マイナビ・リクナビはAPIパートナー契約が必要な場合あり。契約なしではCSVのみ対応。
対応： まずCSV連携で開発。API連携は媒体ごとに個別交渉後に追加
R-04 中 Salesforce 双方向同期の複雑性 コンフリクト・同期タイミングのミスはデータ破損リスク。
対応： U-04で主従関係確定済み。SFマスターとして一方向同期から実装開始
R-05 中 個人情報の取り扱い 学生の個人情報・採用評価情報を扱うため情報漏洩リスクが高い。
対応： 閉じたシステム設計・IP制限・アクセスログ・定期的な脆弱性診断
R-06 中 カレンダー機能の工数 空き枠管理・エージェント含む予約システムは工数が大きい。eeasy廃止の代替として品質担保が必要。
対応： 初期はシンプルな枠予約から実装し段階的に拡張
R-07 中 採用繁忙期の集中アクセス エントリー解禁時期（3月・6月）に大量アクセスが集中する可能性。
対応： 負荷試験の実施とスケールアウト構成の検討
R-08 低 Re就活のAPI提供状況 Re就活のAPI連携情報が未確認。CSV対応のみの可能性あり。
対応： 直接問い合わせて確認。最悪CSVフォールバックで対応
