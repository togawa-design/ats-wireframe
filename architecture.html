サービス別役割
サービスベンダー役割契約
Firebase App Hosting Firebase Next.js アプリのホスティング。SSR・API Routes・静的アセット配信を担当 Google（Firebase）
Firebase Auth Firebase Google OAuth 2.0 認証。staff_role に応じたアクセス制御のトークン発行 Google（Firebase）
Cloud SQL for PostgreSQL Google Cloud メインデータベース。36テーブル・リレーショナル設計・分析クエリを担当 Google（GCP）
Firebase Storage Firebase 書類PDF・QRコード画像のオブジェクトストレージ Google（Firebase）
Next.js API Routes OSS BFF層。LINE Webhook の受け口も兼ねる —（App Hosting上で動作）
サイバーメール（SMTP） NEW 合否連絡・内定通知書のメール送信。Nodemailer 経由で SMTP 接続。接続情報確認待ち NTTコミュニケーションズ（サイバーメール）
LINE Messaging API LINE 応募者とのチャット・リッチメニュー・LIFF LINE（別途契約）
ガルーン REST API サイボウズ 面接枠のスケジュール同期（ATS→ガルーン一方向）。Push Notifications なしのため ATS を正として管理 サイボウズ（ガルーン クラウド版）
Salesforce 確認待ち 採用状況の外部連携（連携仕様はクライアント確認待ち） 未定
すべての Google サービス（Firebase・Cloud SQL）は1つの GCP プロジェクト・1つの請求アカウントで管理。ガルーンはサイボウズ、メールはサイバーメール（NTTコム）との別途契約。

主要データフロー
フロー経路
スタッフ認証 ブラウザ → Firebase Auth（Google OAuth） → Next.js API Routes → Cloud SQL（Staff テーブル照合）
応募者データ取込（CSV） ブラウザ → Next.js API Routes → Cloud SQL（Applicant / ApplicantSource INSERT）
LINE メッセージ受信 LINE Platform → POST /api/webhooks/line（Next.js） → Cloud SQL（LineMessage INSERT）
LINE メッセージ送信 ブラウザ → Next.js API Routes → LINE Messaging API → 応募者 LINE
カレンダー同期（ATS→ガルーン） ブラウザ → Next.js API Routes → Cloud SQL（CalendarSlot 更新） → ガルーン REST API（一方向）
書類PDF生成・保存 ブラウザ → Next.js API Routes → PDF生成 → Firebase Storage → ダウンロードURL返却
自動通知送信 v0.6 Cloud Scheduler（定期実行） → Next.js API Routes（/api/notifications/trigger） → NotificationRule 評価 → LINE / Nodemailer SMTP → NotificationLog INSERT

環境変数（主要なもの）
変数名値の内容管理場所
DATABASE_URLCloud SQL 接続文字列（Cloud SQL Auth Proxy 経由）Firebase App Hosting シークレット
FIREBASE_PROJECT_IDGCP プロジェクト ID.env.local / App Hosting 環境変数
LINE_CHANNEL_SECRETLINE Messaging API チャンネルシークレット（Webhook 署名検証に使用）Firebase App Hosting シークレット
LINE_CHANNEL_ACCESS_TOKENLINE Messaging API アクセストークンFirebase App Hosting シークレット
GOOGLE_CALENDAR_WEBHOOK_TOKENPush Notifications チャンネル検証トークンFirebase App Hosting シークレット
NEXT_PUBLIC_FIREBASE_*Firebase SDK 設定（apiKey, authDomain 等）.env.local（公開可）
LINE_CHANNEL_SECRET と LINE_CHANNEL_ACCESS_TOKEN は絶対にクライアントサイドに露出させないこと。NEXT_PUBLIC_ プレフィックスを付けてはいけない。

デプロイフロー概要
Firebase App Hosting は GitHub リポジトリと直接連携。main ブランチへのマージで本番自動デプロイ。
STEP操作内容環境
1PR 作成feature ブランチから main へ PR。Firebase App Hosting がプレビュー URL を自動生成preview
2レビュー・QAプレビュー URL で動作確認。staging DB を使用staging
3PR マージmain へのマージをトリガーに Firebase App Hosting がビルド・デプロイを開始production
4DB マイグレーションビルドフックで prisma migrate deploy を実行。失敗時はデプロイをロールバックproduction
5ヘルスチェックデプロイ完了後に /api/health へのリクエストで DB 接続・Firebase Auth の疎通を確認production

ローカル起動手順
STEPコマンド内容
1docker compose up -dPostgreSQL コンテナを起動（port: 5432）
2cp .env.example .env.local環境変数ファイルを作成して各値を設定
3npx prisma migrate devマイグレーションを実行して27テーブルを作成
4npx prisma db seed開発用ダミーデータを投入（Staff・マスターデータ等）
5firebase emulators:startFirebase Auth Emulator を起動（port: 9099）
6npm run devNext.js 開発サーバーを起動（port: 3000）
LINE Webhook のローカルテストには ngrok または localtunnel で HTTPS トンネルを作成し、LINE Developers コンソールの Webhook URL に設定する。
